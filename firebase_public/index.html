Reelio helps you see your scrolling habits clearly, so you can understand yourself better.

This page is a guide to how Reelio works in real life. It explains what the app observes, how those signals become risk scores, what each dashboard means, and where the limits are. The model runs on your device and adapts to your own pattern over time.

  • The current data format uses schema version 5 and pipeline version 6.
  • A session is flagged as doom when predicted probability reaches 0.55 or higher.
  • The doom score is explained through 7 readable behavior factors, then aligned with the HMM estimate.
  • Confidence is shown in clear parts: data volume, state separation, stability, and supervision from check-ins.

01 / Why, What, How

A guide first, not a pitch.

Reelio is built to help you notice patterns you usually feel only after the fact. Instead of only counting minutes, it tracks how sessions start, drift, and end. The goal is practical self-understanding: see what pulls you in, what helps you exit, and how those patterns change over time.

Why this exists

Time alone can hide the real story. Two sessions can last the same length but feel very different. Reelio tracks friction, re-entry, and rhythm so risk feels interpretable, not vague.

What is measured

Each reel adds small signals: pace, pauses, exits, rewatches, and context like time of day or environment. These are combined at session and history level before scoring.

How to read this page

This page moves from simple to deep: purpose, architecture, signal layers, model logic, and dashboards. Use it as a reference while reading your own data in the app.

02 / Architecture

From scroll events to insights you can act on.

This is the exact pipeline running behind the app. Read it as five simple questions: what happened, is the data reliable, what state are you in, why that state, and how it is shown to you.

Step 1
InstaAccessibilityService (Kotlin)
First, the Android service records what actually happened during the session in real time. It logs dwell, speed, pauses, rewatches, exits, return gaps, light, posture, charging state, source app, and timing. Think of this as the raw behavior journal.
AccessibilityService SensorManager Room + CSV append-only logging
Step 2
Schema guard and preprocessing (Python)
Second, Python checks that the session data is complete and trustworthy. It validates required columns, removes duplicate or corrupted rows, normalizes labels, and builds derived features such as intent mismatch and fatigue risk when available.
EXPECTED_SCHEMA_VERSION=5 REQUIRED_COLUMNS enforcement dedupe_session_rows preprocess_session
Step 3
ALSE inference core (ReelioCLSE)
Third, ALSE estimates the hidden session state over time. It combines HMM inference, gap-aware transitions, drift checks, contextual priors, and confidence scoring. This is where doom probability is computed, using a fixed threshold of 0.55.
HMM CTMC regime detector confidence breakdown
Step 4
Heuristic anatomy and alignment loop (DoomScorer)
Fourth, a parallel scorer turns risk into plain reasons you can interpret. It uses seven drivers: session length, exit conflict, rapid re-entry, automaticity, dwell collapse, rewatch compulsion, and environment. Weights are re-aligned against HMM output so the explanation stays consistent with the model state.
7 component anatomy dynamic weights intent mismatch amplification fatigue proxy input
Step 5
Dashboard payload and UI projection
Fifth, run_dashboard_payload packages transitions, confidence, drivers, circadian map, feature weights, and session summaries. The dashboards render this same structure so what you see matches what the model actually produced.
PIPELINE_VERSION=6 model_parameters historical_drivers feature_weights

04 / ALSE internals

How ALSE decides, in plain language.

These pillars keep the model stable and interpretable. You do not need to know the math to use this; this section explains what each part protects against and why it matters.

01

Personalized Bayesian Baseline

Learns your personal baseline first, so each score is compared to you.

02

Self-Calibrating Emission Weights

Rebalances feature importance as your behavior pattern shifts.

03

Hierarchical Temporal Memory

Uses short-, mid-, and long-term memory so one unusual day does not distort everything.

04

Continuous-Time Markov Chain

Updates transitions using real time gaps between sessions.

05

Survival Framing

Models how likely you are to stop, not only how long you continue.

06

Regime Change Detector

Flags sudden life-pattern shifts so old behavior does not mislead new predictions.

07

Sparse-Data Guard

When data is limited, it backs off safely instead of making overconfident guesses.

08

Contextual State Priors

Uses session-start context to set stronger starting probabilities.

09

Composite Doom Score

Keeps a readable 7-driver score alongside latent-state inference.

10

Extended Regret Validator

Uses regret labels to correct repeated prediction bias.

11

Confidence Decomposition

Breaks confidence into parts so trust stays transparent and explainable.

12

Dynamic Alignment

Keeps heuristic explanations synchronized with HMM behavior.

06 / Privacy and permission model

Local-first privacy, explained clearly.

Core inference and dashboard generation happen on-device. Permissions exist for specific reasons, and data boundaries are explicit.

Data handling boundaries

  • Core inference works without uploading behavioral telemetry.
  • Model state is stored locally and updated incrementally.
  • run_dashboard_payload replays sessions locally to build dashboards.
  • Reports and delayed-label updates run in local Python.

Permissions and purpose

  • Accessibility service: reads interaction flow during reel sessions.
  • Usage stats: adds entry context such as previous app.
  • Notifications: delivers delayed check-ins and prompts.
  • Sensors: add light, motion, and posture context.

08 / Limitations and known edges

What this system can and cannot claim.

Reelio is useful, but it has limits. Knowing those limits is part of using the insights responsibly.

Accessibility surface drift

Instagram UI changes can alter event shapes and occasionally affect reel boundary quality.

Proxy-based content signals

Content repetition indicators are still proxy-level and not full fingerprint matching.

Cold start latency

First Python invocation in a process can be slower because runtime initialization is required.

Low-data confidence floor

Early sessions rely more heavily on priors by design. Confidence disclosure is required for trust.

Sleep and fatigue are approximations

fatigue_risk is intentionally conservative and should be interpreted as a proxy, not a diagnosis.

09 / Roadmap

What we are improving next.

Near term

  • Better repeat-content detection.
  • More delayed check-ins for stronger calibration.
  • Cleaner drill-down parity between app and export reports.

Mid term

  • Interventions timed using confidence components.
  • Clearer long-term trend narratives from transition drift.
  • Richer pre-session context priors.

Long term

  • Support for more short-form surfaces with local processing preserved.
  • Optional encrypted backup of model state without raw logs.
  • More readable counterfactual guidance for recovery planning.
